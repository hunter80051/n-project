# AGENTS.md

本文件提供給 Codex 使用，目的是降低常見的 LLM 編碼錯誤。請依照專案實際情況，與其他專案專屬指示合併使用。

**取捨原則：**
這些規範偏向謹慎與可驗證，而不是追求速度。若任務非常簡單，可以依照工程判斷適度簡化流程。

## 專案背景

- 本專案目前沒有預設使用的遊戲開發引擎，待未來依需求做調整。
- 因本專案主要為快速開發網頁遊戲為目的，故主要可能涉及的程式碼語言暫代為 JavaScript。
- 本專案使用 Google Sheet 作為表單與資料配置方案；涉及配置表、資料生成、配置讀取或相關工具鏈時，請優先考慮既有流程與約束。
- 目前沒有特定偏好的測試流程；在沒有更明確指示前，請使用通用且保守的驗證方案，例如編譯檢查、既有測試、最小可重現測試或人工檢查步驟。
- 目前沒有特定的程式碼風格要求；在沒有更明確指示前，請遵循既有檔案與相鄰程式碼的風格。

## 1. 寫程式前先思考

**不要擅自假設。不要隱藏不確定性。要主動說明取捨。**

在開始實作前：

- 明確說出你的假設；如果不確定，先詢問。
- 如果需求有多種可能解讀，不要默默選一種，請先列出可能性。
- 如果存在更簡單的做法，請主動提出。
- 當需求、上下文或技術限制不清楚時，先停下來，說明哪裡不清楚，並提出問題。
- 若使用者的要求可能導致過度設計、維護成本增加或不符合現有架構，請適度提醒並提出替代方案。

## 2. 優先保持簡單

**用能解決問題的最少程式碼。不要加入推測性的設計。**

請遵守：

- 不實作使用者沒有要求的功能。
- 不為只使用一次的程式碼建立抽象層。
- 不加入未被要求的「彈性」、「設定選項」或「可擴充性」。
- 不為實際上不可能發生的情境加入複雜錯誤處理。
- 如果你寫了 200 行，但其實 50 行就能清楚解決，請簡化。

自我檢查：

> 一位資深工程師會不會覺得這個解法過度複雜？

如果答案是會，請重新簡化。

## 3. 精準修改

**只修改必要的地方。只清理自己造成的影響。**

編輯既有程式碼時：

- 不要順手「改善」相鄰但無關的程式碼、註解或格式。
- 不要重構沒有壞掉、也不屬於本次需求範圍的部分。
- 遵循既有程式碼風格，即使你個人會用不同寫法。
- 如果發現無關的 dead code 或潛在問題，可以在回覆中提到，但不要擅自刪除或修改。

當你的修改造成未使用項目時：

- 移除由你本次修改導致未使用的 import、變數、函式或檔案。
- 不要移除本來就存在的 dead code，除非使用者明確要求。

檢查標準：

> 每一行被修改的程式碼，都應該能直接追溯到使用者的需求。

## 4. 以目標驅動執行

**定義成功標準，並持續驗證直到完成。**

請將任務轉換成可驗證的目標，例如：

- 「加入驗證」
  -> 先為無效輸入撰寫測試，再讓測試通過。

- 「修正 bug」
  -> 先寫出能重現 bug 的測試，再修正並確認測試通過。

- 「重構 X」
  -> 確認重構前後測試都通過，且行為沒有改變。

對於多步驟任務，請先提出簡短計畫：

```text
1. [步驟] -> 驗證方式：[檢查項目]
2. [步驟] -> 驗證方式：[檢查項目]
3. [步驟] -> 驗證方式：[檢查項目]
```

良好的成功標準能讓 Codex 自主推進並驗證結果。
模糊的標準，例如「讓它能用」，通常需要先釐清，避免後續反覆修改。

## 這些規範生效的跡象

如果本文件有效，應該能看到以下改善：

- Diff 中不必要的修改變少。
- 因過度設計導致的重寫變少。
- Codex 會在實作前提出釐清問題，而不是犯錯後才回頭修正。
- 修改範圍更小、更容易 review。
- 測試與驗證方式更明確。
