# DTTO Friends Dungeon Prototype — MVP 開發計畫

最後更新：2026-07-16
文件狀態：遊戲 MVP、小隊站位／戰鬥表現重構及 Google Sheets 配置同步工具已完成；首次實際 apply 待使用者日後修改 Sheet。

## 1. 任務目標

以 `Game Sample` 的 CLICKPOCALYPSE II 為玩法與技術參考，將目前 GitHub Pages 根目錄的「32 秒點擊遊戲」替換成一款可直接展示的單機網頁 RPG Prototype。

展示版必須具備：

- 固定四人小隊，不製作選角流程。
- 大地圖與地下城之間的循環。
- 地下城共三層；前兩層為程序式房間與走廊，第三層固定 Boss 房。
- 清空當層全部敵人才可使用樓梯。
- 自動探索、自動戰鬥、HP/SP、攻擊動作切換。
- 角色升級時在下方待處理列表加入三選一技能，不自動暫停探索。
- 寶箱拾取與待確認裝備介面；玩家點擊後才替換較佳裝備。
- 卷軸法術；不製作藥水。
- Canvas 遊戲畫面加 DOM 資訊面板。
- 靜態配置與程式碼分離，所有關聯使用穩定 ID。
- 可直接由現有 GitHub Pages workflow 部署。

## 2. 已確認前提與假設

### 2.1 技術與部署

- 現有網站是無 build step 的 `index.html + style.css + script.js`。
- GitHub Pages 從 repository 根目錄部署，因此新遊戲仍維持純靜態檔案。
- 使用原生 ES Modules，不新增 npm、Vite、React 或遊戲引擎。
- 既有 Discord/Codex/Ollama bridge、部署 workflow 及非遊戲檔案不在本次修改範圍。

### 2.2 Google Sheet

- 遊戲執行階段仍讀取 repository CSV，不直接依賴 Google API。
- Phase 7 已加入 OAuth 唯讀 Google Sheets 同步工具；Google Sheet 是人工數值配置來源，只有使用者修改。
- `data/manifest.json` 保存各資料表來源、個別版本與 SHA-256；同步驗證通過後才更新本機 CSV。
- 遊戲不依賴資料列順序；跨表關聯全部使用 `characterId`、`skillId`、`enemyId`、`itemId`、`scrollId` 等 ID。
- Google Sheet、OAuth 桌面用戶端與首次唯讀 API check 均已完成；敏感憑證只保存在已忽略的本機 `secrets/`。

### 2.3 美術與 IP

- 已唯讀檢視指定參考頁：DTTO Friends 是七名角色組成的療癒系角色品牌，視覺以簡單輪廓、粗深色描邊、圓潤比例、低細節及高辨識度配色為主。
- MVP 固定隊伍暫以 Dinu、Hoya、Lynn、Bob 的角色定位作為職能參考。
- 公開部署可能涉及角色名稱與造型授權；在未確認授權前，程式內會讓名稱與 sprite metadata 可配置，並使用低解析度原型圖形／佔位圖，不直接複製參考頁圖片。
- 若後續確認有使用權，可只替換 sprite sheet 與 CSV 名稱，不改遊戲系統。

### 2.4 Ollama 協作

- 本機 Ollama API 已確認可用，模型為 `qwen2.5-coder:7b`。
- Ollama 採適用性派工：只有低耦合、少量明確檔案、驗收條件具體且預期能降低成本的子任務才使用 `ollama_local_agent.py`。
- Codex 負責規格、資料 schema、diff 審查、整合、測試與必要修正。
- Ollama 不操作 Git、Shell、部署或 allowedFiles 以外的檔案。

## 3. 明確不製作的內容

- 城堡系統。
- 地下城 farm／再感染。
- 藥水與藥水欄位。
- 開局選角與角色命名。
- 角色永久死亡、Game Over 或復活流程。
- Prestige、離線補算、成就及 Adventure Points。
- 多人、帳號、雲端存檔、後端 API。
- CLICKPOCALYPSE II 的數百技能、完整裝備與全部職業內容。
- Google Sheets 寫入權限或由 Codex 自動修改 Sheet；目前維持唯讀，數值由使用者手動修改。

## 4. MVP 玩家流程

```text
載入外部配置
  → 大地圖顯示固定小隊與可進入地下城
  → 點擊「進入地下城」
  → 生成第 1 層房間、走廊、敵人與樓梯
  → 小隊自動探索與戰鬥
  → 清空全部敵人後啟用樓梯
  → 第 2 層重新生成
  → 清空後進入第 3 層固定 Boss 房
  → 擊敗 Boss
  → 顯示攻克結算並回到大地圖
  → 下一座地下城提高難度後重複
```

## 5. 遊戲規則

### 5.1 地圖與階段

- Canvas 內部解析度預定 960×540，CSS 等比例縮放。
- 地下城使用 24px 方格，40×22 格可完整放入 960×540 Canvas，不在 MVP 提供設定。
- 大地圖改為執行階段生成的組合地圖：
  - 以固定尺寸的邏輯格組合，繪製時使用 2.5D 菱形投影，形成早期遊戲的拼接地圖感。
  - 地圖資料只保存在 JavaScript 記憶體；同一次開啟期間保留已生成區段，重新整理或重開遊戲後清空。
  - 每攻克或略過一座地下城後，生成通往下一座地下城的新區段；除上一入口周圍的延伸接縫會轉為新道路／草地外，其餘既有道路、草地與場物不重新生成。
  - 鏡頭只顯示地圖的一部分並跟隨隊伍，隊伍的投影位置維持在 Canvas 中央附近。
  - 小隊在大地圖上沿主要道路自動前進，到達下一個地下城入口後顯示地下城情報，玩家可選擇進入或尋找其他地下城。
  - 地下城情報先以展示用資料呈現 1–2 個怪物圖示及一個任務難度標籤；目前不影響敵人生成、掉落、地下城倍率或特殊條件。
  - 選擇進入時維持既有三層地下城流程；選擇尋找其他地下城時保留目前入口並標記為已略過，延展新區段後自動前往新的活動入口。
  - 已略過入口永久保留在記憶體大地圖，不再成為活動目標，不增加已攻克地下城數量，也不提高難度。
  - 已攻克地下城數量與大地圖目的地序號分開保存，為未來完成指定數量後出現魔王城的區域流程保留正確語意；本階段不實作魔王城或區域重置。
- 大地圖分三個繪製層級：
  1. 底層地塊：草地、道路、水面。
  2. 中層場物：樹木、灌木叢、地下城入口、大石頭、小石頭。
  3. 上層角色：固定四人小隊。
- 大地圖區段生成順序：
  1. 確定目前地下城入口與下一個入口的位置。
  2. 以道路格建立保證連通的主要移動路線，並加入少量不影響主路的展示岔路。
  3. 用草地格填出道路周圍的大面積主要地形。
  4. 用水面格填滿目前已生成地形的外圍邊界。
  5. 在道路兩端放置地下城入口；主要道路保持淨空。
  6. 草地內部可形成樹叢與石群；道路交界只少量放置樹木或灌木；岔路終點可少量放置石頭或樹木。
- 大地圖場物使用 45 度視角的簡化圖示與較深色輪廓，先以 Canvas 程式繪圖完成，不新增外部圖片資源。
- 大地圖視覺採使用者提供的 2.5D 島嶼參考方向：
  - 水面是連續底盤，草地與道路以薄菱形拼片覆蓋其上，不製作厚重立方體地塊。
  - 相鄰同類地塊組成不規則大片地形，外緣保留亮色邊線與輕微深色側面以增加高度感。
  - 樹木、灌木、入口與石頭維持直立的 45 度視角圖示，尺寸可跨越單格視覺範圍，但邏輯占位仍為一格。
- 地下城第 1、2 層：
  - 改為 2.5D 等角投影；邏輯仍使用方格尋路，畫面採與大地圖相同的底層地塊／中層場物／上層角色分層。
  - 每層先確定入口房、依序前進的目的房與最終傳送處，再生成 5–7 個不重疊矩形房間及連通走廊。
  - 房間與走廊依前進順序分成 reveal group；開始只顯示入口房，下一段走廊與房間預先生成但保持隱藏。
  - 清空目前已揭露區域後，小隊前往門口；抵達門前才同時揭露下一段走廊、房間、牆壁、場物與敵人。
  - 底層地塊分為可行走的房間／走廊頂面，以及只畫在畫面下緣暴露邊界的外圍厚度側面。
  - 中層場物包含兩種等角軸方向的牆壁與房門、入口樓梯、通往下層的樓梯口與樓梯、Boss 勝利後的傳送魔法陣。
  - 房門只放在預定連通走廊與房間的交界；其餘房間邊界以兩種方向的牆壁場物封閉。
  - 起點位於第一個房間，放置代表抵達本層的入口樓梯。
  - 最後房間的傳送場物依階段決定：第 1、2 層是通往下層的樓梯口；第 3 層是離開地下城的傳送魔法陣。
- 第 3 層：固定單一大型 Boss 房，入口放置樓梯，Boss 擊敗後啟動傳送魔法陣。
- 每次進入樓層保存 seed，方便重現與除錯。

### 5.2 自動探索

- 地圖格類型：牆、地板、走廊、樓梯、入口。
- 小隊使用格狀路徑搜尋前往最近的存活敵人；沒有敵人時前往樓梯。
- MVP 可使用 BFS，因地圖尺寸小且所有可走格成本相同；不因參考遊戲使用 A* 就增加不必要複雜度。
- 小隊保留同行中心負責鏡頭、換房與換層；四名角色各自尋路、避碰並依戰鬥狀態調整站位。

### 5.3 戰鬥

- 固定四人隊伍：Tank、Healer、Caster、Striker。
- 每名角色至少有 HP、SP、attack、defense、speed、attackRange、attackCooldown。
- 敵人靠近後停止移動並進入自動戰鬥。
- 角色與敵人不死亡：
  - 角色 HP 最低鎖定為 1。
  - HP 過低時，Healer 提高治療優先級。
- 攻擊動畫使用兩個 frame：idle / attack，交替顯示並配合短距離位移或閃白。
- MVP 以右側 HP/SP bar、攻擊幀與事件列呈現數值變化；Canvas 浮動數字延後至美術強化階段。
- SP 會隨時間恢復；角色技能與卷軸可消耗 SP 或卷軸次數。

### 5.4 升級選技

- 全隊共享擊殺經驗；角色達到門檻時升級。
- 升級時在 Canvas 下方待處理列表加入技能選項，模擬繼續運作。
- 從該角色尚未取得的技能中選出最多三項。
- 點選技能後立即加入角色技能列表；只有玩家手動按下暫停時才停止遊戲。
- 同一角色連升多級時，同時只顯示最舊的一筆技能選擇；後續升級先依發生順序排隊，完成前一筆後再依已學技能產生下一筆，避免重複選項。
- 若可選技能不足三項，顯示實際數量；全部學完則給予固定屬性成長。

### 5.5 裝備

- MVP 裝備欄位：weapon、armor。
- 敵人依 loot table 機率掉落裝備。
- 若新裝備的評分高於目前裝備，加入右下角待確認清單，不立即替換。
- 玩家點擊整張裝備通知後才套用數值並顯示「角色：舊裝備 → 新裝備」提示；通知不可略過。
- 裝備只提供可配置的 attack、defense、maxHp、maxSp 修正。

### 5.6 卷軸

- MVP 至少三種：Fire Burst、Chain Spark、Healing Light。
- 卷軸由敵人掉落並顯示堆疊數量。
- 玩家透過 DOM 按鈕主動施放；沒有有效目標時按鈕 disabled。
- 卷軸效果、SP 成本、傷害／治療倍率及圖示索引由 CSV 控制。
- 不製作自動施放、無限卷軸或藥水互動。

### 5.7 小隊站位、碰撞與戰鬥職責

- 每名角色與敵人都有獨立世界座標，不再以單一 `partyPosition` 加固定畫面偏移代表整隊。
- 所有存活角色與敵人之間必須保留碰撞距離，不得互相重疊；移動後的位置必須位於已揭露的可走地板內。
- 角色與怪物的碰撞外框取樣點都必須維持在可走地板，不能只檢查中心點；靠近牆面時必須保留足以容納模型落腳點的內縮距離。
- 前緣牆與前向門框必須在角色之後繪製，使靠近牆邊的角色仍被牆正確遮擋，不呈現站在牆外的錯覺。
- 不同房間／走廊揭露群組的可走地塊若相鄰，群組交界必須永久保留邊界牆；只有預定連通門洞的中央一格可開口，揭露下一區後原房間牆面也不可消失。
- 非戰鬥狀態包含大地圖移動、進出地下城、換房、換區與前往樓梯／傳送處：
  - 保留一個小隊同行中心，負責鏡頭、旅程進度與區域觸發。
  - 四名角色在同行中心附近使用鬆散隊形與獨立移動，允許自然交換站位，但不可離開可走地形。
  - 大地圖角色若落在不可走地塊、連續移動受阻或距離同行中心過遠，必須自動重新集結到同行中心附近的安全空位，不可留在上一個地下城入口或地圖邊緣。
  - 地下城走廊必須容納四名獨立角色跟隨同行中心通過；若現有寬度造成角色碰撞半徑或模型視覺超出地板，需擴寬走廊且維持既有不穿越其他房間／走廊的生成規則。
  - 三格寬走廊與房間交界保留中央門框；門框繪製時直接延伸左右兩段實牆，確保門洞兩側沒有空缺，同時不得把四人導航通道縮成單格瓶頸。
- 戰鬥狀態不限制角色維持同行隊形：
  - 近戰角色主動接近目標，盡可能在不重疊的前提下貼近攻擊。
  - 遠程角色避開敵人近身，優先移往較空曠且仍在攻擊距離內的位置，保持安全距離後攻擊。
- 角色戰鬥類型以外部角色配置明確區分近戰與遠程；遠程攻擊另區分物理與魔法投射物。
- 角色選敵優先序：
  1. 正在攻擊自己的敵人。
  2. 正在攻擊任一遠程隊友的敵人。
  3. 距離及攻擊條件最合適的其他敵人。
- 敵人會保留目前攻擊目標，讓角色能依「正在攻擊誰」進行保護與反擊判斷。
- 遠程普通攻擊不立即結算傷害；必須生成可飛行的投射物，命中目標後才結算。物理與魔法使用不同輪廓、軌跡與顏色。

### 5.8 掉落寶箱與待確認裝備

- 敵人產生任一掉落時，在敵人倒下位置生成寶箱；物品與卷軸先保存在寶箱內，不立即加入小隊。
- 任一角色接觸寶箱後視為全隊取得內容物，寶箱消失並依內容更新事件列。
- 若目前已揭露區域沒有存活怪物，寶箱產生後 3 秒仍未被任一角色接觸，直接視為全隊拾取並收進背包，避免探索流程卡住。
- 較佳裝備不立即穿戴，而是加入主畫面右下角的待確認裝備清單。
- 每張待確認卡片顯示角色、欄位、舊裝備、新裝備及主要數值差異；整張卡片皆可點擊。
- 只有點擊裝備卡片後才套用裝備並重新計算角色數值；多件掉落可同時排隊，且不可略過。
- 未處理的待確認裝備可跨樓層保留，離開該次地下城時清除。
- 任一裝備套用後，其他待確認提案必須依最新裝備重新比較：重新分配給仍可受益的隊員，若全隊目前裝備評分皆不低於該物品則移除提案。
- UI 不得顯示或允許「目前評分 → 更低評分」的換裝選項。
- 技能升級與裝備更換共用同一個待處理列表，依建立時間由舊到新排列，可交錯顯示。
- 裝備通知整張元件皆可點擊；點擊後立即換裝，不再提供「更換／略過」二選一，也不可略過。
- 換裝成功後的短暫彈跳通知固定顯示在 Canvas 遊戲畫面的右下角，不使用整個瀏覽器視窗的 fixed 定位。
- 技能項目在同一列表內顯示角色名稱及技能選項，點擊其中一個技能後完成該項目。
- 待處理列表位於 Canvas 下方右欄，不得浮在卷軸按鈕、角色面板或 Canvas 上攔截點擊。
- 冒險記錄位於 Canvas 下方左欄，改為縱向列表並由最新事件置頂。

### 5.9 戰鬥與卷軸特效

- 遠程角色攻擊以飛行投射物表現；物理投射物使用較實體的短尾軌跡，魔法投射物使用發光核心與粒子尾跡。
- 點擊卷軸時，由符合效果的角色作為施法者：傷害卷軸優先由遠程攻擊角色施放，治療卷軸優先由 Healer 施放。
- 卷軸按鈕在數量及可用狀態未改變時維持同一 DOM 元素；一次單擊即施放一次，不可要求連點兩下。
- 傷害卷軸從施法者位置飛向敵人或在敵群間連鎖；治療卷軸從施法者位置擴散至隊友，特效完成時套用效果。
- 角色頭像與 Canvas 模型維持程式繪圖，參考 DTTO FRIENDS 的圓潤比例、粗深色描邊與高辨識度配色重新設計，但不直接臨摹官方素材。

## 10. Ollama 派工判斷

只有符合以下條件才派工：

1. 可限制在一至數個明確 allowedFiles。
2. 驗收條件具體且結果容易由 diff 判斷。
3. 與其他模組耦合低，不需要架構或根因判斷。
4. 屬於常數、文案、CSS、小型純函式、重複性轉換或單檔案局部邏輯。

多模組重構、程序地圖核心、狀態機、跨檔案整合、除錯根因、安全、Git 與部署不交給 Ollama。沒有合適子任務時由 Codex 直接完成並說明原因，不做形式性派工。

每次派工後：

1. Codex 查看變更檔案與 diff。
2. 檢查是否遵守 ID 索引及 MVP 範圍。
3. 執行語法／資料驗證。
4. 不符合規格或回傳格式失敗時，保留派工紀錄後可由 Codex 直接接手；只有修正成本明顯較低時才重派一次。

## 11. 測試案例

### 資料

- manifest 所有本機表格存在。
- 每張表必填欄完整。
- 主鍵不重複。
- character 的 skill ID、enemy 的 loot table、loot 的 item／scroll、dungeon 的 enemy／boss 全部可解析。
- 數值欄不可為 NaN，機率介於 0–1。

### 地圖

- 大地圖起點到下一個地下城入口存在連續道路。
- 每次完成地下城後只追加新區段；除上一入口周圍的延伸接縫外，先前道路、草地與場物維持不變。
- 大地圖主要道路沒有樹木、灌木或石頭阻擋。
- 草地與道路外圍存在連續水面邊界。
- 鏡頭移動時小隊維持在畫面中央附近，畫面只繪製視野周圍格位。
- 重新建立 `GameSimulation` 後，大地圖回到初始區段。
- 地下城開始時只顯示入口房；未揭露走廊、房間、牆壁、場物與敵人不可見且不參與戰鬥。
- 目前區域敵人全滅後，小隊會前往下一扇門，抵達門前才揭露下一個 reveal group。
- 所有房門位於房間與走廊的有效連接邊界，且具有符合等角軸線的兩種面向之一。
- 房間與走廊暴露的下緣具有外圍厚度側面；非下緣不重複堆疊厚度。
- 第 1、2 層最後區域顯示下層樓梯口；第 3 層 Boss 擊敗後顯示並啟用傳送魔法陣。
- 第 1、2 層至少生成指定數量房間。
- 所有房間可由起點到達。
- 樓梯位於可走格。
- 第 3 層只有 Boss 房且存在 Boss。
- 清怪前不可下樓，清怪後可下樓。

### 戰鬥

- 小隊會前往敵人並停止在攻擊範圍。
- 攻擊時 idle／attack frame 有可見變化。
- 角色 HP 不低於 1。
- SP 消耗與恢復正確。
- 敵人死亡後發放 XP 並可能掉落。

### 成長與 UI

- 升級項目顯示 1–3 個有效技能，且不會自動暫停模擬。
- 選擇後技能立即生效並移除該項目；未選擇時探索、寶箱逾時計時與戰鬥仍繼續。
- 較佳裝備進入待確認清單；點擊更換後才套用並顯示提示，較差裝備不建立提案。
- 卷軸數量為 0 時按鈕 disabled。
- 攻克第三層後回到大地圖並提升 dungeon run。
- 抵達大地圖入口後顯示 1–2 個怪物圖示與單一任務難度，且情報不改變實際戰鬥內容。
- 選擇尋找其他地下城後，舊入口與既有地形保持不變，小隊改往新入口移動。
- 連續略過多座地下城不增加 dungeon run 或難度；進入目前活動入口時載入該入口記錄的地下城。
- 多筆較佳裝備可排入右下角待確認清單；未點擊前角色裝備與數值不變。
- 點擊裝備通知元件後才更新裝備與數值；裝備通知不提供略過。
- 換裝成功提示顯示在 Canvas 右下角，不遮擋下方冒險記錄、待處理列表或右側角色面板。
- 接受較高分裝備後，所有剩餘提案重新計算受益角色；不得保留評分下降的提案。
- 多名角色同時升級時，所有技能選擇皆加入共用待處理列表；與裝備項目依建立順序混合排列，最舊項目置頂。
- 同一角色連升多級時不會同時產生重複技能選項；依序完成後仍可處理每次升級。

### 站位、碰撞與特效

- 非戰鬥時四名角色在同行中心附近獨立移動，換房與換層仍能由全隊完成。
- 四名角色通過所有程序走廊時不會因通道過窄卡住，角色碰撞範圍與模型落點不超出已揭露地板；若不符合則調整生成走廊寬度後重新驗證。
- 戰鬥時近戰靠近目標、遠程遠離近身敵人並尋找空曠攻擊位置。
- 任何更新步驟後，存活角色與敵人均位於已揭露可走地板，且任兩個戰鬥單位不重疊。
- 角色與怪物的中心及碰撞外框取樣點都位於已揭露地板；前緣牆會遮擋角色下半部，不會顯示在牆外。
- 每扇房門兩側皆存在可見牆面，且四名角色仍可通過門口，不因牆面補齊形成單格瓶頸。
- 每個逐區揭露階段，已揭露地板與尚未揭露可走格之間都沒有黑色牆面缺口；揭露下一區後，原房間與新走廊相鄰的邊界牆仍保留，只開放門洞中央。
- 近戰與遠程皆依「攻擊自己 → 攻擊遠程隊友 → 其他合適敵人」選敵。
- 遠程傷害只在投射物命中後結算，物理與魔法投射物視覺可辨識。
- 敵人掉落先生成寶箱，任一角色接觸後才取得物品或卷軸。
- 已揭露區域沒有存活怪物時，未接觸寶箱會在 3 秒內自動收取，之後隊伍能繼續前往房門或樓梯。
- 三種卷軸皆可看到從合適角色位置開始的施法特效，且效果在特效命中／擴散時生效。
- 卷軸與裝備通知皆以一次單擊觸發；按下與放開之間即使遊戲持續更新，也不會因按鈕被替換而遺失事件。
- 部署後的 CSS 與 ES Module 入口使用版本參數更新快取；重新開啟驗證連結時必須載入本次版本。

## 12. 風險與處理方式

| 風險 | 處理 |
| --- | --- |
| Ollama 一次產出過多導致品質不穩 | 小批次 allowedFiles、每批語法檢查與 diff 審查。 |
| CSV 外部資料載入失敗 | 啟動畫面顯示明確錯誤，不進入半初始化狀態。 |
| 程序地圖偶發不連通 | 先建立連通骨架再加隨機連線，並執行可達性檢查。 |
| 自動戰鬥卡住 | 尋路失敗時重選目標；超時後重新定位到最近可走格。 |
| 展示時間過長 | CSV 控制敵人數、HP、XP 與速度，預設每層約 1–2 分鐘。 |
| DTTO 角色公開使用授權不明 | 名稱、配色、sprite 全部可替換；MVP 不嵌入參考頁原圖。 |
| GitHub Pages 不支援本機 file:// fetch | README 使用本機 HTTP server；正式 Pages 不受影響。 |

- 最終 sprite sheet 是否由 Image 2.0 產生、人工繪製或由授權素材提供。

這些事項不影響目前以可替換資料與 sprite metadata 建立 Prototype。

## 15. MVP 完成結果

- `python validate_project.py` 通過：7 個 ES Module、8 張 CSV、HTML 引用、數字欄、主鍵與外鍵全部有效。
- 地圖生成以 100 個 seed 驗證房間數、全房間連通、入口到樓梯路徑及 Boss 房。
- 無頭模擬在 517 個固定步長內完成三層地下城，四名角色皆保持 HP ≥ 1，且技能 ID 無重複。
- 卷軸測試確認 Fire Burst 造成 288 總傷害、Healing Light 恢復 208 總 HP。
- 瀏覽器實際完成 Cozy Cellar 三層與 Boss，執行 16 次升級選技後回到大地圖，下一站切換為 Midnight Workshop。
- 瀏覽器 console 沒有 error 或 warning。
- 600px viewport 時 body 無水平溢出，主畫面與 sidebar 都切換為單欄。

### 大地圖第一階段重構

- 新增執行階段 `Map` 暫存的延展式世界地圖；重建 `GameSimulation` 即回到初始地圖。
- 連續生成 8 個區段測試通過：共 2,434 格、139 格主路，所有主路逐格連通且沒有非地下城場物阻擋。
- 測試結果包含 182 道路、1,389 草地、863 水面，以及樹木、灌木、地下城、大石頭、小石頭五類場物。
- 無頭流程在 59 個世界移動步長後抵達入口，並在 517 個地下城步長後回到大地圖；地圖由 1 區延展為 2 區且開始前往下一入口。
- 瀏覽器確認隊伍保持 Canvas 中央、地圖隨隊伍移動、抵達前隱藏入口按鈕、抵達後顯示按鈕，且可正常進入地下城第 1 層。
- 大地圖視覺依參考圖調整為薄菱形島嶼、連續水面底盤、亮色地形外緣及直立深色描邊場物。

### 地下城第一階段重構

- 地下城改為與大地圖相同的 2.5D 三層繪製：地塊頂面與外圍厚度、牆門／樓梯／傳送場物、敵人與小隊。
- 100 個 seed 生成測試通過；共檢查 290 個 `axisX` 房門與 204 個 `axisY` 房門，所有門前觸發點與最終出口均可達。
- 每層開始只揭露入口房；未揭露地塊、牆門、場物與敵人不顯示，也不參與尋路、戰鬥或卷軸效果。
- 無頭完整流程在 510 個固定步長內完成三層，逐區揭露序列、第 1／2 層下層樓梯及第 3 層 Boss 傳送陣皆正常。
- 瀏覽器確認前緣低牆、兩種等角牆門、房門揭露時機、入口樓梯、樓層厚度與 Boss 傳送魔法陣均可見。

### 小隊站位與戰鬥表現第二階段重構

- 四名角色與所有敵人改為獨立座標；大地圖與地下城完整流程均未出現地板外落點，最小戰鬥單位距離維持約 0.68 格。
- 非戰鬥由同行中心負責鏡頭、門與樓梯觸發；門內側使用固定集結槽位，三格寬走廊可供四人繞行且不會因前排占位卡住。
- 近戰攻擊時平均距離約 1.09 格，遠程約 3.13 格；物理與魔法投射物皆在命中後結算傷害。
- 敵人保留攻擊目標，近戰與遠程均通過「攻擊自己 → 保護遠程隊友 → 其他合適敵人」優先序測試。
- 掉落先生成寶箱，角色接觸後才取得內容；較佳裝備在點擊下方待處理卡片前不改變角色裝備與數值，離開地下城時清除未處理提案。
- 三種卷軸皆由適合角色位置發出可見特效；治療擴散、火焰飛行與閃電連鎖均完成瀏覽器檢查。
- 100 個地圖 seed 共檢查 35,124 個三格寬走廊 footprint，未出現窄化、走廊交叉或入口到樓梯不可達。
- 10 個連續地下城難度／seed 組合皆完成三層流程；瀏覽器桌面與窄畫面 console 均無 error 或 warning。

### 大地圖地下城自由選擇追加

- `dungeonRun` 只計算實際攻克數；另以目的地序號追蹤完成或略過後產生的新入口。
- 每個入口保存固定的展示用情報，從哥布林、黑暗妖精、幽靈、野獸、獸人中顯示 1–2 張圖片，並顯示一般、困難、大遷徙、災變或特殊條件其中一項假難度。
- 情報面板縮為約 300px 並固定在主畫面右下角；怪物圖示改用五格正方形 sprite，難度文字依一般黑、困難綠、大遷徙橘黃、災變紅、特殊條件紫顯示。
- 假情報不影響實際敵人、倍率、掉落或地下城規則；用途是呈現玩家選擇攻略目標的企劃方向。
- 連續略過 6 座地下城的流程測試通過；舊入口均保留、狀態改為 skipped、完成數維持 0，第 7 座入口仍為活動目標。
- 瀏覽器實際略過 Cozy Cellar 後前往 Midnight Workshop，情報由幽靈／野獸／一般切換為野獸／獸人／特殊條件，並能正常進入新地下城。
- 600px viewport 無水平溢出，情報、圖示與兩個操作按鈕皆保持可見；console 無 error 或 warning。
